# v0.18.7 System Administration and Configuration Plan

## Summary

Expand System Settings into production-facing operational configuration. This slice keeps the existing settings API and adds configurable schedule-health, notification, scheduler, and audit retention values.

## Key Changes

- Extend `GET /settings` and `PATCH /settings` with schedule, notification, scheduler, and retention fields.
- Store all values in `system_settings` and restore defaults through seeded reset.
- Audit each changed setting with `SYSTEM_SETTING_UPDATED`.
- Apply settings to schedule-health classification, notification due windows, and email enablement.
- Add a lightweight local-process scheduler that checks the DB setting before running due checks.
- Split the Settings page into Portal Defaults, Schedule Health, Notifications, and Audit and Retention sections.

## Test Plan

- Run standard SDLC gates and Docker Compose validation.
- Verify backend setting validation, audit logging, schedule-health windows, notification windows, email disablement, and scheduler no-op behavior.
- Run regression to confirm System Admin settings update, Auditor write denial, seeded reset defaults, and existing workflows.

## Assumptions

- Settings are global portal settings, not user preferences.
- Database settings are authoritative after reset; environment variables remain startup fallback values for notification settings.
- The scheduler is suitable for the Docker VM pilot and is not intended to replace later queue-backed scheduling.
